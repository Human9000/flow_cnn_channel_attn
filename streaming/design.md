# 流式推理与整段推理数值一致方案

## 1. 问题陈述

在嵌入式流式推理场景中，需要将训练时使用的长序列（如 1000 点）拆分为小块（如 25 点）进行逐块推理。**核心要求**：流式推理（逐块）的输出必须与 PyTorch 整段推理的输出严格一致（允许极小的浮点舍入误差）。

## 2. 数学基础

### 2.1 卷积的局部性

对于任意卷积层（包括空洞卷积、步长>1），其输出 yt*y**t* 仅依赖于输入的一个有限邻域 xt−R+1,...,xt*x**t*−*R*+1,...,*x**t*，其中 R*R* 为该层的**感受野**（Receptive Field）。因此，整个网络是一个**有限脉冲响应（FIR）系统**，天然支持滑动窗口分解。

### 2.2 因果性与非因果性
- 不需要特殊的因果卷积，标准卷积CNN就可以，只需要添加一个onnx转换器，就可以实现因果卷积的流处理效果
- **训练阶段**：PyTorch 使用整段长序列（如 1000 点），卷积核默认采用 `padding=0`，这意味着序列边界处会进行零填充。但训练时我们通常按完整序列计算，不关心边界。
- **推理阶段**：若将长序列切分为不重叠的块（如 25 点），块边界处会丢失历史上下文。**但**如果我们在块之间保留一定的重叠（即缓存历史尾巴），则块与块之间的边界信息得以保留，从而保证输出与整段序列一致。

### 2.3 关键结论

只要满足以下条件，流式推理与整段推理的结果 **数学上完全等价**：

1. 每一层的输入块在时间轴上的拼接方式，与整段序列中对应位置的输入完全一致。
2. 每一层的输出块经过裁剪后，恰好对应整段序列中相同位置的输出。
3. 所有中间状态（缓存）在所有时间步上正确传递。

## 3. 缓存长度计算（通用公式）

对于卷积层参数：

- 核大小 K*K*
- 步长 S*S*
- 空洞率 D*D*

定义 **有效核大小**（即感受野跨度）：

Keff=(K−1)×D+1*K**e**ff*=(*K*−1)×*D*+1

为了确保块边界处不丢失信息，需要在当前块之前拼接的历史点数量为：

Lcache=Keff−1=(K−1)×D*L**c**a**c**h**e*=*K**e**ff*−1=(*K*−1)×*D*

**特别说明**：

- 当 Lcache=0*L**c**a**c**h**e*=0 时（如 K=1*K*=1），无需缓存。
- 对于多层网络，每一层独立维护自己的缓存，缓存长度可能不同。
- 步长 S*S* 不影响缓存长度（因为我们在块内已经使用步长 S*S* 进行卷积，但为了保证输出块长度等于输入块长度，需要确保卷积后时间维不变；若 S>1*S*>1，则需要通过 padding 或调整块大小来保持维度一致。但通常流式推理中，我们采用 S=1*S*=1 的卷积以保持时间分辨率，若有下采样（如池化），则块大小需相应调整。）

## 4. 实现方案

有两种等效的实现路径，均能保证数值一致性。

### 4.1 路径一：ONNX 图变换（推荐）

**流程**：

1. **训练**：PyTorch 使用长序列（如 1000 点），模型为纯卷积结构，无任何缓存逻辑。导出标准 ONNX（动态轴支持可变长度）。
2. **图变换**：使用 `onnx_graphsurgeon` 对 ONNX 模型进行手术，为每个卷积层插入：
   - 一个额外的输入张量（缓存状态，形状为 `[1, C, L_cache]`）
   - 一个 `Concat` 节点，将缓存拼接到当前输入前。
   - 一个 `Slice` 节点，从拼接后的张量中取出末尾 `L_cache` 个点作为新的缓存输出。
3. **推理**：ONNX Runtime 按块（25 点）执行，每次传入当前块和上一轮缓存，输出预测块和更新后的缓存。

**数值等价性**：由于图变换只是在输入侧拼接了历史数据，而原始模型的计算逻辑未变，因此每一块的输出等同于整段序列中对应位置的局部计算，拼接后等于整段推理。

### 4.2 路径二：PyTorch 包装器导出

**流程**：

1. **定义包装器**：在 PyTorch 中定义一个 `StreamingWrapper` 模型，接收 `x`（当前块）和 `state_in`（缓存列表），内部执行拼接并调用原始模型，输出 `y` 和 `state_out`。
2. **导出 ONNX**：将包装器导出为 ONNX，输入包含 `x` 和所有 `state_in`，输出包含 `y` 和所有 `state_out`。
3. **推理**：与路径一相同。

**数值等价性**：包装器在 `forward` 中执行的操作与图变换完全一致，因此同样保证等价。

### 4.3 两种路径的对比

| 方面             | 路径一（图变换）         | 路径二（包装器）       |
| :--------------- | :----------------------- | :--------------------- |
| 修改位置         | ONNX 图结构              | PyTorch 模型代码       |
| 对训练代码的影响 | 无                       | 需额外定义包装器       |
| 导出后的 ONNX    | 完全标准，无自定义算子   | 完全标准，无自定义算子 |
| 灵活性           | 高，可自动化处理任意模型 | 需手动适配每一层       |

**推荐**：若模型结构固定，路径二更直观；若需通用工具，路径一更佳。

## 5. 数值一致性证明（数学推导）

设整段输入序列为 X=[x0,x1,…,xT−1]*X*=[*x*0,*x*1,…,*x**T*−1]，整段推理输出为 Y=f(X)*Y*=*f*(*X*)，其中 f*f* 为模型函数（含所有卷积、池化等）。

现按块大小 B*B* 将 X*X* 分割为 N*N* 个不重叠的块：

X=[X0,X1,…,XN−1],Xi=[xiB,…,x(i+1)B−1]*X*=[*X*0,*X*1,…,*X**N*−1],*X**i*=[*x**i**B*,…,*x*(*i*+1)*B*−1]

对于每一层卷积，其输出依赖于输入的一个有限窗口。若我们将块 Xi*X**i* 前拼接上足够的历史缓存（长度为 Lcache*L**c**a**c**h**e*），则拼接后的输入为：

X~i=[historyi,Xi]*X*~*i*=[history*i*,*X**i*]

其中 historyihistory*i* 为上一轮缓存的 Lcache*L**c**a**c**h**e* 个点，且该历史值恰好等于整段序列中 Xi*X**i* 前面的 Lcache*L**c**a**c**h**e* 个真实值（在稳态下）。

经过卷积层后，输出为：

Y~i=g(X~i)*Y*~*i*=*g*(*X*~*i*)

由于卷积的线性移位不变性，Y~i*Y*~*i* 的最后 B*B* 个点与整段序列中对应位置的输出 Yi=f(Xi)*Y**i*=*f*(*X**i*) 的对应部分完全相等（证明：卷积计算在局部窗口内，拼接后的局部窗口与整段序列中的局部窗口完全一致）。

逐层传递缓存，等价于在整段序列上滑动窗口。因此，最终拼接后的输出 [Y0,Y1,…,YN−1][*Y*0,*Y*1,…,*Y**N*−1] 与整段推理输出 Y*Y* 逐点相等（仅浮点运算顺序可能带来微小差异，但通常 < 1e-5）。

## 6. 验证方法

### 6.1 验证流程

1. 随机生成长度为 T（如 1000）的测试输入张量。
2. 使用 PyTorch 模型进行整段推理，得到输出 `y_full`。
3. 使用导出的流式 ONNX 模型（或 PyTorch 包装器）按块（B=25）进行流式推理，得到 `y_stream`。
4. 计算绝对误差和相对误差。

### 6.2 预期结果

- **最大绝对误差** < 1e-5（单精度浮点误差范围）
- **相对误差** < 1e-4

### 6.3 调试要点

若出现显著误差，可能原因：

- 缓存长度计算错误（未包含空洞卷积的跨步）。
- 缓存未正确更新（如未取末尾缓存点，而是取了开头）。
- 块边界处存在零填充（应为真实历史值，而非零）。
- 下采样层（池化、步长>1）导致块大小与缓存长度不匹配（需相应调整块大小）。

## 7. 注意事项

### 7.1 模型结构限制

- 禁止使用全局池化（如 `AdaptiveAvgPool1d(1)`）、`Flatten` + `Linear` 等压缩时间维的操作。
- Batch Normalization 在推理时使用固定的 running 统计量，不影响流式拆解（因为它是逐点操作）。

### 7.2 下采样与块大小

- 如果模型包含步长 >1 的卷积或池化，输出块长度会缩短。此时需要调整块大小，使得输出块长度仍为期望值（如 25）。通常做法是：在输入块中增加适当数量的额外历史点，确保输出长度不变。

### 7.3 冷启动

- 系统启动时，缓存尚未填满，应初始化为零。前几个块的输出可能包含边界效应，需在验证时排除（或等待预热完成）。

## 8. 总结

通过**缓存拼接**（状态显式化）的方式，流式推理可以在数学上精确复现整段推理的结果。无论是通过 ONNX 图变换还是 PyTorch 包装器，只要保证每一层缓存了足够的有效历史上下文（长度为 (K−1)×D(*K*−1)×*D*），并正确传递和更新，最终数值将完全一致。

该方案已在多种心电图（ECG）分割模型上验证成功，流式推理误差与整段推理的浮点误差相当（< 1e-6）。采用此方法，即可在嵌入式设备上实现高精度、低延迟的实时推理。

## 9. 当前 U-Net 的时间对齐计算

前面的 `(K-1)×D` 是单个卷积的局部缓存公式，不能直接作为含池化、
上采样和跳连的整个 U-Net 的延迟。当前模型需要同时计算张量长度、采样
相位和每条分支在原始输入时间轴上的感受野。

设输入长度为 `T`，各层长度为：

```text
res1 = T - 2
p1   = floor((T - 2) / 2)
res2 = p1 - 2
p2   = floor((p1 - 2) / 2)
d1   = 2*p2 - 2
d2   = 2*d1 - 2
```

第一次跳连 `align_center(res2, d1)` 的长度差为 2 或 3，因此左侧固定
裁剪 1 个低采样点，相当于原始输入时间轴上的 2 点偏移。

第二次跳连 `align_center(res1, d2)` 的左裁剪量取决于输入长度模 4：

| `T mod 4` | `res1 - d2` | 左裁剪点数 |
|---:|---:|---:|
| 0 | 12 | 6 |
| 1 | 13 | 6 |
| 2 | 10 | 5 |
| 3 | 11 | 5 |

沿主干继续传播感受野后，输出位置相对其输出索引需要的最右输入点随
`index mod 4` 分别为 13、12、15、14，最大右上下文为 15 点。它与
最终输出长度损失 14 是两个不同概念，不能用 14 直接代表实时延迟。

当前转换器采用逐张量未完成尾部状态，不再使用单一的 16 点整网输入
历史。Conv 保存尚不足以形成下一个卷积窗口的输入尾部；AveragePool
保存不足一个池化窗口的余数；skip 保存已由编码器产生、但尚未与解码器
对齐消费的队列。这种定义保留标准网络的 valid Conv、池化和跳连坐标。

## 10. 转换期符号执行与动态状态接口

转换器不能再把缓存理解成固定长度的零张量。转换时先按 ONNX 拓扑模拟
一次时间维执行，为每个张量维护以下信息：

```text
TensorTimeState = {
    scale,          # 相对原始输入的采样倍率
    phase,          # 当前采样相位
    left_context,   # 生成当前张量需要的左历史
    right_context,  # 尚未到达的右上下文
    cache_size,     # 跨块后仍需保留的元素数
    live_across_chunk
}
```

传播顺序必须与图的拓扑顺序相同：Conv 更新感受野；Pool 同时更新倍率和
相位；Resize 降低倍率；Slice 更新坐标原点；Add/Concat 先把所有输入换算
到相同时间坐标。尤其是 `align_center` 导出的动态 Slice，必须根据两条
分支长度计算裁剪起点，不能只根据卷积核推算缓存。

只有满足 `live_across_chunk=true` 且 `cache_size>0` 的张量才是状态。
每个状态必须同时生成：

1. `cache_*_in` 图输入；
2. 与当前张量拼接的 `Concat`；
3. 更新历史的 `Slice`；
4. `cache_*_out` 图输出。

不能给块内临时张量创建无意义状态，也不能创建未参与后续计算的 Slice。
当前逐层无重算执行计划共产生 8 组状态：4 个 k=3 Conv 尾部、2 个
AveragePool 余数和 2 个 skip 等待队列。k=1 的输出 Conv 缓存大小为 0，
因此不创建状态。

### 10.1 当前动态接口

```text
input                    float32 [1, 1, chunk_time]
*_cache_in               float32 [1, C, cache_time]

output                   float32 [1, 4, output_time]
*_cache_out              float32 [1, C, cache_time_out]
```

`chunk_time`、每个 `cache_time` 和 `output_time` 都是动态维。所有状态冷
启动时传入零长度缓存。缓存张量本身的动态时间维就是有效长度，不再额外
传递 `cache_count_in/out`，也不再用 `Shape/Gather/Sub/Slice` 重选有效缓存。
转换图使用 `Shape/Gather/Mod/Slice` 模拟必要的动态长度：Conv 状态稳定为 2 点；
Pool 状态在 0～1 点间变化；skip 队列按解码器本块实际产出长度消费。

skip 使用 `cache_in.shape[2] == 0` 判断首次执行，并分别删除 `align_center`
的左裁剪量：de1 为 1 个低尺度点，de2 为 6 个原尺度点。合法首次调用后
skip 队列始终非空，因此无需额外的 `started_in/out` 状态。后续块不重复
裁剪，尚未匹配的 skip 尾部作为状态传到下一块。

这里不把重复片段封装成自定义 ONNX 算子，因为自定义域会降低 ONNX
Runtime、Netron 和嵌入式后端的兼容性。转换器代码中的 `StateBuilder`
负责组合生成标准节点；导出的 ONNX 保持展开形式，但只保留必要运算。

### 10.2 转换期启动分析与宿主调度

ONNX Runtime 的 Conv 和 AveragePool 不能在输入短于 kernel 时产生零长度
输出，因此流式 ONNX 不接收不能产生输出的调用。转换器在导出前模拟标准
ONNX 的时间维执行，沿拓扑传播 Conv、Pool、Resize、Slice 和残差 Add 的
长度，自动搜索两个约束：

```text
startup_samples：冷启动时使最终输出首次非空的最小输入长度
steady_samples：已有状态后保证最终输出增加的最小新输入长度
```

当前分析结果：

| 模型 | startup_samples | startup_outputs | steady_samples | steady_outputs |
|---|---:|---:|---:|---:|
| U-Net | 14 | 2 | 4 | 4 |
| ResNetV2 | 11 | 1 | 1 | 1 |

调度由宿主完成：第一次调用至少输入 `startup_samples`，后续每次至少输入
`steady_samples`，并检查输出非空。流式 ONNX 本身不包含 `If`、空输出
分支或原始输入 pending 状态。转换结果对象会返回上述四个值，ONNX 的
metadata 也保存：

```text
streaming.startup_samples
streaming.startup_outputs
streaming.steady_samples
streaming.steady_outputs
streaming.host_scheduled = true
```

因此 ResNetV2 可以在首次输入 11 点后逐点调用；U-Net 因两次下采样，
首次输入 14 点后仍需每次至少提供 4 个新点。若采集端每次只有 1 点，宿主
应在调用 ONNX 前累计到相应阈值。

## 11. 静态 C actor runtime

ONNX 继续使用前述宿主调度模式；C 版采用独立的数据流 actor runtime，
不区分 warmup/step 入口。调用方每次可推入 1 个采样，节点只有在输入
token 和输出 FIFO 空间都满足算子条件时才进入可执行队列：

```text
push input token
  -> 唤醒 input 的直接消费者
  -> ready 节点进入队列
  -> 节点饱和消费现有输入并产生输出
  -> 更新各 reader 生命周期并释放已被全部消费者读取的 token
  -> 唤醒输出的直接消费者
  -> 直到可执行队列为空
```

没有最终输出是合法状态：U-Net 前 13 个采样、ResNetV2 前 10 个采样只会
累积内部 FIFO。达到网络感受野后，输出自然出现，不需要 `If`、零长度
Conv 或预热专用函数。

### 11.1 静态内存

每条 tensor 边使用一个多消费者环形 FIFO。每个消费者有独立 read
sequence；只有所有消费者都越过某个 token 后，该槽位才能被生产者覆盖。
转换器离线模拟逐点调度，得到每条 FIFO 的最大同时存活 token 数，并建立
同时存活冲突图。无冲突的 tensor 允许共享 16 字节对齐的 arena offset。

`export_streaming_c.py` 生成：

```text
c_generated/unet_stream.c/.h
c_generated/unet_stream_memory_plan.json
c_generated/resnetv2_stream.c/.h
c_generated/resnetv2_stream_memory_plan.json
```

JSON 列出每个 tensor 的通道数、FIFO 容量、字节数、固定 offset、生产者、
消费者、生命周期事件和共享地址关系。运行时不调用 malloc/free；模型实例
的 context 包含静态 arena、FIFO 指针状态、节点状态和可执行队列。权重是
生成 C 文件中的 `static const float`，位于只读区。

当前生成结果：U-Net arena 为 2896 字节，ResNetV2 arena 为 1808 字节。
逐点输入 1000 点时，C 输出与标准 ONNX 的最大误差分别为 `8.94e-08` 和
`5.96e-08`。
